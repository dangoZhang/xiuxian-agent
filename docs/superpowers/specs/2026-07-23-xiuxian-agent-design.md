# 修仙agent：设计规格

状态：已确认方向，等待书面规格复核  
日期：2026-07-23  
项目名与 GitHub 仓库名：`修仙agent`

## 1. 产品定义

`修仙agent` 是一部由大模型实时驱动、可游玩的修仙小说。玩家通过自然语言参与世界；天道生成并维护规则，命运依据因果生成事件，修士 Agent 在神识预算内观察、谋划、调用工具和战斗。所有结果沿时间轴沉淀为可阅读、可追溯的故事。

大模型 API 是运行游戏的强制依赖。未完成模型配置与连通性校验时，游戏不能开局。项目不提供离线剧情、模板事件或无模型降级模式。

## 2. 技术选择

统一使用 **TypeScript**。

- Web：Vue 3、Vite、TypeScript。
- Server：Node.js、Hono、TypeScript。
- 模型调用：Vercel AI SDK，接收玩家配置的 OpenAI-compatible API。
- 数据校验：Zod。
- 状态：服务端事件溯源；客户端只持有投影状态。
- 测试：Vitest；浏览器端关键流程使用 Playwright。
- 包管理：pnpm workspace。

选择 TypeScript 的理由：前后端共享事件协议、工具定义和 Zod schema；Vue 适合构建高密度互动小说界面；Node 原生支持流式响应和并行调度；Vercel AI SDK 已覆盖结构化输出、流式文本与工具调用。引入 Python 作为主业务语言会增加跨进程协议和部署成本。

仓库结构：

```text
修仙agent/
├── apps/
│   ├── web/          # Vue 界面
│   └── server/       # Hono API、会话与模型调用
├── packages/
│   ├── protocol/     # 共享类型、Zod schema、SSE 事件
│   ├── engine/       # 时间轴、状态归约、神识调度、战斗裁决
│   └── prompts/      # 四类角色提示词与工具说明
├── vendor/
│   └── litellm/      # 可选 Git submodule，多供应商统一网关
├── docs/
├── docker-compose.yml
└── pnpm-workspace.yaml
```

LiteLLM 仅负责把多家模型统一为 OpenAI-compatible API。游戏只有一个模型协议，不维护多套供应商 SDK。玩家也可直接填写任何兼容端点。

## 3. 核心世界模型

### 3.1 四种权力

| 来源 | 职责 | 可做 | 不可做 |
|---|---|---|---|
| 天道 | 生成世界规则并解释规则 | 创建灵气、境界、寿元、因果和天劫规则 | 直接决定单次事件胜负 |
| 命运 | 根据当前状态生成下一件有意义的事 | 提出机缘、危机、相遇、代价和期限 | 改写已生效天道规则 |
| 玩家 | 用自然语言表达行动、对话和意图 | 询问、承诺、欺骗、修炼、移动、战斗 | 直接修改世界状态 |
| 修士 | 作为 Agent 自主观察、计划、调用工具和对战 | 在认知范围内行动并形成记忆 | 读取超出神识和可见性的信息 |

天道和命运都由大模型生成结构化提案。服务端校验提案后写入状态。修士行为由大模型工具调用驱动。所有数值变化只经引擎归约器生效。

### 3.2 时间轴

游戏时间使用整数 `tick`，一个 tick 表示一天。界面显示为“道历 X 年 X 月 X 日”。

时间轴使用优先队列。每次只推进到最近的有效事件，不逐日空转。以下情况触发循环：

1. 玩家提交一句话或一个行动。
2. 已安排事件到期。
3. 当前结果产生新的因果事件。
4. 修士仍有未结算的并行意图。

循环顺序固定：

```text
读取最近事件
→ 天道校验边界
→ 命运生成事件
→ 等待玩家对话或行动
→ 相关修士在神识预算内并行行动
→ 引擎统一结算
→ 写入时间轴并安排后续事件
→ 若无玩家决策点则跳到下一事件
```

每条时间轴记录包含：

```ts
type ChronicleEntry = {
  id: string
  tick: number
  source: 'heaven' | 'fate' | 'player' | 'cultivator' | 'combat'
  status: 'proposal' | 'resolved'
  actorIds: string[]
  causeIds: string[]
  visibility: string[]
  text: string
  structuredPayload: unknown
  cost: { sense: number; qi: number }
}
```

`causeIds` 构成因果链。界面可从当前段落追溯到规则、对话和先前事件。

## 4. 神识：玩法资源与并行限制

神识同时限制认知和计算。

每名修士拥有：

- `senseMax`：一个时间窗口内的总预算。
- `focusSlots`：可同时执行的意图数。
- `perceptionRange`：可读取的地点、人物与历史范围。
- `memoryDepth`：送入模型的个人记忆条数。

工具调用消耗固定神识：观察 1、推演 2、传音 1、施法 2、追踪 2、破阵 3。修士只能调用预算内且满足可见性的工具。

服务端使用加权信号量调度：

- 单个修士的并行量不得超过 `focusSlots`。
- 全局并行量由 `MAX_AGENT_CONCURRENCY` 限制。
- 同一战斗回合的意图并行生成，全部到齐后统一结算。
- 调用顺序不影响胜负。

神识耗尽后，该修士只能等待下一个时间窗口恢复。突破境界可提高神识上限和并行槽位。

## 5. 修士 Agent 与工具

每名修士拥有公开状态、隐藏目标、人格、关系、功法、个人记忆和当前计划。模型只接收该修士可见的信息。

首版工具集固定为：

```text
observe(target)       观察人物、地点或异象
remember(query)       检索个人记忆
move(destination)     前往相邻地点
speak(target, words)  当面交谈或传音
cultivate(method)     修炼功法或恢复资源
useTechnique(name, target) 施展功法
guard(target)         防御或保护
scheme(goal)          形成跨时间计划
```

模型不能直接写状态。每个工具先校验神识、位置、资源和规则，再生成领域事件。非法调用返回明确原因，并保留在该修士私有推演记录中。

## 6. 对战

战斗按回合运行，每回合包含四步：

1. 参战修士并行观察可见战况。
2. 各自通过工具提交一个主意图，可按 `focusSlots` 附加辅助意图。
3. 引擎依据速度、功法克制、灵力、伤势、环境和带种子随机数统一结算。
4. 大模型根据结构化结果写出各自视角，时间轴写入一条战斗事实。

模型决定策略与表达；引擎决定命中、伤害、消耗、状态和死亡。战斗支持撤退、求饶、结盟、欺骗和环境利用。

## 7. 一局游戏

### 7.1 开局

玩家必须完成模型设置：

- Base URL
- API Key
- 模型名称
- 全局最大并发
- 天道、命运、修士是否使用同一模型；高级设置可分别指定

服务端执行一次结构化输出与工具调用校验。校验成功后才能创建世界。

玩家输入一句身世。天道据此生成本局规则；命运生成出生事件；三名关键修士生成各自目标与关系。

### 7.2 核心资源

- 气血：归零会重伤或死亡。
- 灵力：施法和修炼消耗。
- 神识：限制观察、工具调用和并行意图。
- 寿元：时间推进的长期压力。
- 因果：承诺、伤害和恩情形成的债。
- 境界：炼气、筑基、金丹。

### 7.3 可玩内容

首个完整版本包含：

- 一次由炼气至金丹的完整人生。
- 三个区域、两个宗门、六名持续行动的修士 Agent。
- 四类功法路线：剑、丹、阵、魔。
- 关系、因果债、受伤、死亡、突破和天劫。
- 至少三类结局：证道、陨落、入魔；具体文本由本局状态生成。
- 自动存档、手动导出和导入。
- 同一存档可继续游玩；不承诺模型输出的逐字复现。

## 8. 对话与界面

桌面布局采用三栏：

- 左：可缩放时间轴、事件节点和因果连线。
- 中：小说正文流、角色对话、玩家输入框和建议行动。
- 右：角色状态、神识槽、关系、功法与正在运行的 Agent。

移动端按“正文 / 时间轴 / 角色”切换，输入框固定在底部。

来源必须通过颜色、边框、标题印记和排版共同区分：

| 来源 | 视觉 |
|---|---|
| 天道 | 暗金双线诏令框，竖排印记“天道”，庄严短句 |
| 命运 | 紫灰虚线谶语框，标题带卦象，含期限与代价 |
| 玩家 | 青玉色对话框，保留原始输入 |
| 修士 | 墨色单线人物框，显示姓名、境界和神识消耗 |
| 战斗 | 朱红角标与断裂框线，先展示事实再展示叙事 |

界面显示当前并行调用：每个修士对应一枚“神念”，运行时沿因果线移动；神识槽实时扣减。动画尊重 `prefers-reduced-motion`。

## 9. 模型协议

所有模型输出采用 Zod schema 约束。四类请求分别定义 prompt 和输出结构：

- `HeavenRuleSet`：世界常量、禁止项、境界与资源规则。
- `FateEventProposal`：触发条件、参与者、风险、期限、候选后果。
- `PlayerIntent`：从玩家原话提取目标、对象和行动，不改写原话。
- `CultivatorTurn`：思考摘要、工具调用和可见发言。

服务端通过 SSE 发送：`call.started`、`text.delta`、`tool.called`、`call.completed`、`entry.resolved`、`game.paused`。

模型调用失败时立即暂停时间轴。玩家可重试或修改模型配置。结构化输出校验失败时自动重试一次；再次失败则暂停。系统不生成替代剧情。

API Key 只保存在服务端进程内存或服务端环境变量中，不写入浏览器存储、存档、日志或 Git。

## 10. 服务接口

```text
POST /api/model/test              校验配置、结构化输出和工具调用
POST /api/sessions                创建带模型配置的临时会话
POST /api/games                   创建世界
GET  /api/games/:id               获取当前投影
POST /api/games/:id/commands      提交玩家原话
GET  /api/games/:id/events        SSE 事件流
POST /api/games/:id/advance       推进到下一决策点
GET  /api/games/:id/export        导出存档
POST /api/games/import            导入存档
```

前端不能直接调用模型供应商。所有请求经服务端执行，便于保护密钥、校验工具和控制并发。

## 11. 数据与存档

首版使用 SQLite 保存世界状态和时间轴。模型密钥不入库。

权威数据只有：

- 初始天道规则。
- 已结算领域事件。
- 修士私有记忆事件。
- 当前优先队列。
- 随机种子和引擎版本。

页面状态由事件归约生成。导出格式包含 schema 版本，导入时严格校验。损坏或版本不兼容的存档直接拒绝，并指出字段。

## 12. 错误处理

- 缺少或无法连接模型：停留在设置页。
- 模型超时、限流或余额不足：暂停当前 tick，保留未结算意图，允许重试。
- Agent 返回非法工具：拒绝该工具并将原因返回同一 Agent；最多重新规划一次。
- 并发预算不足：任务排队，不超发请求。
- 服务进程中断：从最后一条已结算事件恢复，未结算调用全部作废。

## 13. 测试与验收

单元测试覆盖：

- 优先队列与按需时间推进。
- 事件归约和因果链。
- 神识扣费、单体槽位与全局并发上限。
- 战斗意图顺序无关性。
- 可见性过滤与私有记忆。
- 四类模型 schema。

集成测试使用可控的测试模型端点，覆盖开局、对话、事件、并行 Agent、战斗、暂停重试、存档导入。

浏览器验收：

1. 未配置可用 API 时无法开局。
2. 玩家输入自然语言后，能看到四类来源清晰分隔。
3. 时间轴只推进到有事件的时刻。
4. 六名修士可自主行动，实际并发不超过配置和神识限制。
5. 至少一场战斗由多个修士 Agent 并行提交意图并完成结算。
6. 一局可从出生推进至明确结局。
7. 桌面和移动端均可完整操作。
8. 导出的存档可重新导入并继续。

## 15. 实施边界

首版不做多人联机、交易市场、社区模组编辑器、语音、3D 场景、移动原生应用或模型训练。它们不影响“由大模型驱动、时间轴循环、四权叙事、神识并行和修士对战”的核心完成度。
