# 修仙agent：设计规格

状态：已批准
日期：2026-07-23  
项目名与 GitHub 仓库名：`修仙agent`

## 1. 产品定义

`修仙agent` 是一部由大模型实时驱动、可游玩的修仙小说。玩家通过自然语言参与世界；天道生成并维护规则，命运依据因果生成事件，修士 Agent 在神识预算内观察、谋划、调用工具和战斗。所有结果沿时间轴沉淀为可阅读、可追溯的故事。

大模型 API 是运行游戏的强制依赖。未完成模型配置与连通性校验时，游戏不能开局。项目不提供离线剧情、模板事件或无模型降级模式。

## 2. 技术选择

第一方业务代码统一使用 **TypeScript**。

- Web：Vue 3、Vite、TypeScript。
- Server：Node.js、Hono、TypeScript。
- 模型调用：Vercel AI SDK，接收玩家配置的 OpenAI-compatible API。
- 数据校验：Zod。
- 状态：服务端事件溯源；客户端只持有投影状态。
- 测试：Vitest；浏览器端关键流程使用 Playwright。
- 包管理：pnpm workspace。

选择 TypeScript 的理由：前后端共享事件协议、工具定义和 Zod schema；Vue 适合构建高密度互动小说界面；Node 原生支持流式响应和并行调度；Vercel AI SDK 已覆盖结构化输出、流式文本与工具调用。引入 Python 作为主业务语言会增加跨进程协议和部署成本。

仓库结构：

```text
修仙agent/
├── apps/
│   ├── web/          # Vue 界面
│   └── server/       # Hono API、会话与模型调用
├── packages/
│   ├── protocol/     # 共享类型、Zod schema、SSE 事件
│   ├── engine/       # 时间轴、状态归约、神识调度、战斗裁决
│   └── prompts/      # 四类角色提示词与工具说明
├── vendor/
│   └── litellm/      # 可选外部 Git submodule，多供应商统一网关
├── docs/
├── docker-compose.yml
└── pnpm-workspace.yaml
```

LiteLLM 仅负责把多家模型统一为 OpenAI-compatible API，不参与游戏逻辑或默认启动。游戏只维护一个模型协议，不再增加供应商适配层。玩家也可直接填写满足最低能力要求的兼容端点。

## 3. 核心世界模型

### 3.1 四种权力

| 来源 | 职责 | 可做 | 不可做 |
|---|---|---|---|
| 天道 | 生成世界规则并解释规则 | 创建灵气、境界、寿元、因果和天劫规则 | 直接决定单次事件胜负 |
| 命运 | 根据当前状态生成下一件有意义的事 | 提出机缘、危机、相遇、代价和期限 | 改写已生效天道规则 |
| 玩家 | 控制一名主角修士，用自然语言表达行动、对话和意图 | 询问、承诺、欺骗、修炼、移动、战斗 | 直接修改世界状态或交给自主 Agent 代替决策 |
| 修士 | 作为 Agent 自主观察、计划、调用工具和对战 | 在认知范围内行动并形成记忆 | 读取超出神识和可见性的信息 |

天道和命运都由大模型生成结构化提案。服务端校验提案后写入状态。玩家原话先解析为 `PlayerIntent`，再进入与其他修士相同的工具、可见性、资源和战斗校验链。其他修士行为由大模型工具调用驱动。所有数值变化只经引擎归约器生效。

### 3.2 天道规则 DSL

天道只能生成受限的 `HeavenRuleSet`，不能生成代码或任意公式。规则集包含：

- `constants`：灵气倍率、寿元基数、境界阈值等允许覆盖的数值常量。
- `modifiers`：从引擎注册表选择的加减、倍率、上限和下限修正。
- `triggers`：从注册表选择的突破、受伤、立誓、杀戮、时间和地点触发器。
- `effects`：从注册表选择的资源变化、状态增删、事件安排和关系变化。
- `descriptions`：供界面展示的规则原文，不参与计算。

服务端以 Zod 校验规则，再编译为引擎可执行表。规则集保存 `schemaVersion`、递增 `ruleVersion` 和内容哈希。天道只在开局或已注册的“纪元变更”触发器发生时提出新版本；新版本从下一个事件序列生效，不能改写历史。

### 3.3 时间轴

游戏时间使用整数 `day`，界面显示为“道历 X 年 X 月 X 日”。同一天内使用 `phase` 和全局单调递增的 `sequence` 排序。阶段固定为 `rule → fate → player → agent → resolve → narrate`。优先队列排序键为 `(day, priority, insertionSequence)`，同优先级仍稳定排序。

时间轴使用优先队列。每次只推进到最近的有效事件，不逐日空转。以下情况触发循环：

1. 玩家在决策点提交一句话或一个行动。
2. 已安排事件到期。
3. 当前结果产生新的因果事件。
4. 修士计划到期或感知到与自身有关的变化。

循环按事件需要选择处理器：

```text
读取最近事件
→ 仅在规则缺失、纪元变更或规则冲突时调用天道
→ 仅在命运钩子到期、因果链需要外部变故或事件队列为空时调用命运
→ 仅在玩家决策点等待输入；收到输入后直接继续当前循环
→ 唤醒事件相关修士，在神识预算内并行行动
→ 引擎统一结算
→ 写入时间轴并安排后续事件
→ 若无玩家决策点则跳到下一事件
```

修士唤醒条件固定为：身处事件感知范围、被事件点名、个人计划到期、目标与事件冲突、正在参与战斗。其他修士不调用模型。

一次 `advance` 在遇到玩家决策点、游戏结束、模型错误或完成 32 个微步时停止。达到 32 个微步会暂停并报告调度状态，防止无限因果循环。

每条时间轴记录包含：

```ts
type ChronicleEntry = {
  id: string
  day: number
  phase: 'rule' | 'fate' | 'player' | 'agent' | 'resolve' | 'narrate'
  sequence: number
  source: 'heaven' | 'fate' | 'player' | 'cultivator'
  kind: 'rule' | 'event' | 'dialogue' | 'action' | 'combat' | 'breakthrough'
  actorIds: string[]
  causeIds: string[]
  visibility: string[]
  text: string
  structuredPayload: unknown
  cost: { sense: number; qi: number }
}
```

`causeIds` 构成因果链。界面可从当前段落追溯到规则、对话和先前事件。战斗属于 `kind`，叙事来源仍是四权之一。

提案和未完成调用只存在于一次内存事务。整轮模型调用、工具校验和结算全部成功后，服务端原子写入已结算的 `ChronicleEntry`；任一步失败则丢弃整轮草稿，并从相同 revision 重试。存档不包含提案或半成品意图。

## 4. 神识：玩法资源与并行限制

神识同时限制认知和计算。

每名修士拥有：

- `senseMax`：一个行动窗口内的总预算。
- `focusSlots`：可同时占用的模型调用与意图槽位数。
- `perceptionRange`：可读取的地点、人物与历史范围。
- `memoryDepth`：送入模型的个人记忆条数。

一个行动窗口是普通事件中的一次激活，或战斗中的一个回合。窗口开始时神识恢复至 `senseMax`。模型规划调用预留 1 点神识和 1 个 `focusSlot`；读取工具立即执行，行动工具在收集完成后统一结算。

工具成本固定：

| 工具 | 成本 | 类型 |
|---|---:|---|
| `observe` | 1 | 读取 |
| `remember` | 1 | 读取 |
| `move` | 1 | 行动 |
| `speak` | 1 | 行动 |
| `cultivate` | 2 | 行动 |
| `useTechnique` | 2 | 行动 |
| `guard` | 1 | 行动 |
| `scheme` | 2 | 行动 |

修士只能调用预算内且满足可见性的工具。非法工具不扣工具成本；已经完成的规划调用仍消耗 1 点。整轮事务失败时所有临时扣费回滚。

服务端使用加权信号量调度：

- 单个修士同时运行的模型调用不得超过 `focusSlots`。
- 全局并行量由 `MAX_AGENT_CONCURRENCY` 限制。
- 同一战斗回合的意图并行生成，全部到齐后统一结算。
- 调用顺序不影响胜负。

神识耗尽后，该修士在当前窗口不能继续读取或提交行动。突破境界可提高神识上限和并行槽位。

## 5. 修士 Agent 与工具

每名修士拥有公开状态、隐藏目标、人格、关系、功法、个人记忆和当前计划。模型只接收该修士可见的信息。

首版工具集固定为：

```text
observe(target)       观察人物、地点或异象
remember(query)       检索个人记忆
move(destination)     前往相邻地点
speak(target, words)  当面交谈或传音
cultivate(method)     修炼功法或恢复资源
useTechnique(name, target) 施展功法
guard(target)         防御或保护
scheme(goal)          形成跨时间计划
```

模型不能直接写状态。每个工具先校验神识、位置、资源和规则，再生成领域事件。非法调用返回明确原因，并允许该修士在同一事务内重新规划一次。

## 6. 对战

修士战斗状态固定包含 `hp`、`qi`、`senseMax`、`speed`、境界、状态列表和功法列表。功法包含 `power`、`qiCost`、标签、克制标签、目标范围与结构化效果；天道只能通过规则 DSL 修改允许的系数。

战斗按回合运行，每回合包含六步：

1. 引擎冻结回合开始时的战斗状态快照。
2. 玩家用自然语言提交主角意图；其他参战修士并行观察并通过工具提交一个主意图，可按 `focusSlots` 附加辅助意图。
3. 全部存活参战者都提交后关闭收集。模型超时或缺失意图会暂停本回合，不自动代选动作。
4. 引擎先结算防御和位移，再依据速度、功法克制、灵力、伤势与环境计算攻击；同阶段按 `(speed DESC, cultivatorId ASC)` 稳定排序，伤害与状态在阶段结束时批量提交。
5. 随机结果由 `hash(battleSeed, round, actorId, actionIndex, effectId)` 独立生成，不消费与遍历顺序相关的随机流。
6. 大模型根据结构化结果写出各自视角，时间轴写入已结算战斗事实。

模型决定策略与表达；引擎决定命中、伤害、消耗和状态。撤退是行动，成功条件由速度、阻拦状态和环境计算。`hp <= 0` 且没有生效的护命效果时死亡；主角死亡进入陨落结局。求饶、结盟、欺骗和环境利用都经工具与规则校验。

## 7. 一局游戏

### 7.1 开局

玩家必须完成模型设置：

- Base URL
- API Key
- 模型名称
- 全局最大并发
- 天道、命运、修士是否使用同一模型；高级设置可分别指定

服务端执行结构化输出、工具调用与流式文本三项校验。全部成功后才能创建世界。

玩家输入一句身世。天道据此生成本局规则；命运生成出生事件；三名关键修士生成各自目标与关系。

### 7.2 核心资源

- 气血：归零会重伤或死亡。
- 灵力：施法和修炼消耗。
- 神识：限制观察、工具调用和并行意图。
- 寿元：时间推进的长期压力。
- 因果：承诺、伤害和恩情形成的债。
- 境界：炼气、筑基、金丹。

### 7.3 可玩内容

首个完整版本包含：

- 一次由炼气至金丹的完整人生。
- 三个区域、两个宗门、六名持续行动的修士 Agent。
- 四类功法路线：剑、丹、阵、魔。
- 关系、因果债、受伤、死亡、突破和天劫。
- 至少三类结局：证道、陨落、入魔；具体文本由本局状态生成。
- 自动存档、手动导出和导入。
- 同一存档可继续游玩；不承诺模型输出的逐字复现。

## 8. 对话与界面

桌面布局采用三栏：

- 左：可缩放时间轴、事件节点和因果连线。
- 中：小说正文流、角色对话、玩家输入框和建议行动。
- 右：角色状态、神识槽、关系、功法与正在运行的 Agent。

移动端按“正文 / 时间轴 / 角色”切换，输入框固定在底部。

来源必须通过颜色、边框、标题印记和排版共同区分：

| 来源 | 视觉 |
|---|---|
| 天道 | 暗金双线诏令框，竖排印记“天道”，庄严短句 |
| 命运 | 紫灰虚线谶语框，标题带卦象，含期限与代价 |
| 玩家 | 青玉色对话框，保留原始输入 |
| 修士 | 墨色单线人物框，显示姓名、境界和神识消耗 |
| 战斗 | 朱红角标与断裂框线，先展示事实再展示叙事 |

界面显示当前并行调用：每个修士对应一枚“神念”，运行时沿因果线移动；神识槽实时扣减。动画尊重 `prefers-reduced-motion`。

## 9. 模型协议

所有影响状态的模型输出采用 Zod schema 约束。纯叙事文本可以流式输出，但不能修改状态。请求类型包括：

- `WorldGenesis`：开局地点、宗门与角色槽位。
- `CultivatorProfile`：人格、目标、关系、功法与公开信息。
- `HeavenRuleSet`：受限规则 DSL。
- `HeavenRulePatch`：纪元变更后的新规则版本。
- `FateEventProposal`：触发条件、参与者、风险、期限、候选后果。
- `PlayerIntent`：从玩家原话提取目标、对象和行动，不改写原话。
- `CultivatorTurn`：简短行动理由、工具调用和可见发言。
- `NarrationRequest`：只消费已结算事实，返回流式小说文本。

兼容端点必须支持 JSON schema 结构化输出、工具调用和流式文本；多修士并行由服务端发起多个独立请求，不要求供应商原生并行工具调用。开局校验逐项验证三种能力，不满足即拒绝开局。

服务端通过 SSE 发送：`call.started`、`text.delta`、`tool.called`、`call.completed`、`entry.resolved`、`game.paused`。每条 SSE 都携带 `callId`、`source`、`actorId` 和 `sequence`，并发文本不会串流。

模型调用失败时立即暂停时间轴。玩家可重试或修改模型配置。结构化输出校验失败时自动重试一次；再次失败则暂停。系统不生成替代剧情。

API Key 只保存在服务端进程内存或服务端环境变量中，不写入浏览器存储、存档、日志或 Git。

## 10. 服务接口

```text
POST /api/model/test              校验配置、结构化输出、工具调用和流式文本
POST /api/sessions                创建带模型配置的临时会话
POST /api/games                   创建世界
GET  /api/games/:id               获取当前投影
POST /api/games/:id/commands      提交玩家原话
GET  /api/games/:id/events        SSE 事件流
POST /api/games/:id/advance       推进到下一决策点
GET  /api/games/:id/export        导出存档
POST /api/games/import            导入存档
```

前端不能直接调用模型供应商。所有请求经服务端执行，便于保护密钥、校验工具和控制并发。

## 11. 数据与存档

首版使用 SQLite 保存世界状态和时间轴。模型密钥不入库。

权威数据只有：

- 初始天道规则。
- 已结算领域事件。
- 修士私有记忆事件。
- 当前优先队列。
- 随机种子和引擎版本。

页面状态由事件归约生成。导出格式包含 schema 版本，导入时严格校验。损坏或版本不兼容的存档直接拒绝，并指出字段。

## 12. 错误处理

- 缺少或无法连接模型：停留在设置页。
- 模型超时、限流或余额不足：丢弃当前内存事务，暂停在原 revision，允许整轮重试。
- Agent 返回非法工具：拒绝该工具并将原因返回同一 Agent；最多重新规划一次。
- 并发预算不足：任务排队，不超发请求。
- 服务进程中断：从最后一条已结算事件恢复；内存中的未结算调用不存在于存档。

## 13. 测试与验收

单元测试覆盖：

- 优先队列与按需时间推进。
- 事件归约和因果链。
- 神识扣费、单体槽位与全局并发上限。
- 战斗意图顺序无关性。
- 可见性过滤与私有记忆。
- 所有影响状态的模型 schema。

集成测试使用仅存在于测试进程的可控模型夹具，覆盖开局、对话、事件、并行 Agent、战斗、暂停重试、存档导入。该夹具不进入生产依赖或构建产物。

浏览器验收：

1. 缺少 API，或端点未通过结构化输出、工具调用、流式文本三项校验时，创建游戏接口返回明确错误。
2. 固定验收场景执行三条玩家命令后，时间轴至少各出现一条 `heaven`、`fate`、`player`、`cultivator` 来源记录，且 DOM 具有不同边框类、标题印记和可读标签。
3. 从第 1 日安排第 10 日事件时，中间不生成空日记录；队列直接弹出第 10 日。
4. 并行场景中至少两个修士模型调用的时间区间重叠；峰值并发不超过 `MAX_AGENT_CONCURRENCY`，每名修士神识支出不超过 `senseMax`，同时调用不超过 `focusSlots`。
5. 同一战斗快照和意图集合以不同输入顺序结算，最终状态哈希一致。
6. 一局可从出生推进至证道、陨落或入魔之一；结局记录包含完整因果链。
7. 390×844 与 1440×900 视口均能完成模型设置、开局、三轮对话、一次战斗和存档导出，无横向溢出。
8. 导出再导入后的规则、角色、队列和时间轴状态哈希一致，并可继续结算一轮。
